# 기획_Phase2 (LOOP WORLD — 살아 숨쉬는 세계)

> 전제: Phase 0(창발) + Phase 1(관찰/개입/필) 완료.
> Phase 2 목표: **정적인 환경을 리듬 있는 환경으로.**
> 지금 세계는 강우가 전역 상수라 조건이 늘 똑같다. 여기에 시간(계절)과 공간(기후대·지형)의
> 변주를 넣으면, 플레이어가 아무것도 안 해도 세계가 흥망을 반복한다.

---

## 핵심 판단: 왜 환경 먼저인가

원래 Phase 2 후보는 셋(GPU 스케일 / 환경 / 종 추가). 셋 다 하지 않는다.

| 후보 | 성격 | 판정 |
|---|---|---|
| **환경 시스템** | 3종 그대로 극적 변화. 밸런싱 리스크 낮음 | **Phase 2 본체** |
| GPU 백만 셀 | 규모의 장관. 게임성 향상 아님 | 선택 트랙 A (마지막) |
| 종 추가 | 깊이는 늘지만 밸런싱 지옥 | 선택 트랙 B (신중히) |

환경 시스템은 **새 종 없이** 게임을 살린다. 계절 하나만 넣어도
겨울 대멸종·봄 부활이라는 리듬이 생겨 관찰 재미가 폭발한다.

---

## 1. 환경 시스템 (Phase 2 본체)

복잡도·임팩트 순으로 세 단계. **1-1만 해도 성과가 보인다.**

### 1-1. 계절 (전역 사인파) — 제일 쉽고 임팩트 최대

전역 시간을 1년 주기로 돌려서 강우·성장을 변조한다.

```csharp
// day = 누적 틱 수, yearLength = 예: 120틱(=120일)
float seasonPhase = (day % yearLength) / (float)yearLength; // 0~1
float season = Mathf.Sin(seasonPhase * 2f * Mathf.PI);      // -1~1

// 강우·성장을 계절로 변조
float rainNow   = cfg.rainfall   * (1f + season * cfg.seasonRainAmp);
float growthNow = cfg.growthRate * (1f + season * cfg.seasonGrowAmp);
```

- 여름(season>0): 강우·성장 ↑ → 개체군 폭발.
- 겨울(season<0): 강우·성장 ↓ → 대량 아사. 살아남은 개체가 봄에 재점화.
- **이 리듬 하나가 게임을 완전히 바꾼다.** 매년 흥망이 반복되고, 개입 타이밍(겨울 전에 심을지)이 의미를 가진다.

구현: `SimulationConfig`에 `yearLength, seasonRainAmp, seasonGrowAmp` 추가.
`LocalRules`에서 상수 대신 계절 변조값 사용. UI에 현재 계절 표시.

### 1-2. 기후대 (위도별 기본 조건) — 공간 다양성

세계에 위도를 부여해 위/아래는 춥고 중앙은 따뜻하게.

- 토러스 y-wrap을 유지하려면 위도를 코사인으로:
```csharp
// y 끝과 시작이 이어지므로, 두 개의 온대 밴드가 생기게
float lat = Mathf.Cos((y / (float)H) * 2f * Mathf.PI); // 중앙=1(따뜻), 끝=-1(추움)
float tempFactor = Mathf.Clamp01(0.5f + 0.5f * lat);   // 0~1
```
- `tempFactor`를 성장률에 곱함 → 추운 띠는 성장 느림(툰드라), 따뜻한 띠는 무성(정글).
- 계절 × 위도 조합: 극지방은 겨울에 완전 동결, 적도는 사철 생산. **공간적으로 다른 운명**이 갈린다.

> 위도를 넣으면 확산으로 "따뜻한 곳 → 추운 곳" 침투가 생겨서, 계절 따라 생명대가
> 북상·남하하는 게 관찰된다. 이게 히트맵에서 진짜 예쁘게 보임.

### 1-3. 물 흐름 & 지형 (강·호수) — 제일 복잡, 마지막

높이맵(elevation)을 추가해 물이 낮은 곳으로 흐르게.

- 새 배열 `float[] Elevation` (고정, 세계 생성 시 노이즈로 1회 생성).
- 확산 대신 **방향성 흐름**: 각 셀의 water 일부를 이웃 중 가장 낮은 곳으로 이동.
```csharp
// 물은 낮은 곳으로. 최저 이웃 하나로 흘려보냄
int lowest = FindLowestNeighbor(x, y);
float flow = water[i] * cfg.flowRate;
water[i] -= flow;
water[lowest] += flow;
```
- 결과: 저지대에 물이 고여 **호수·강**이 자연 발생. 강가에만 생명이 몰리고 고지대는 사막.
- 이건 사실상 새 시뮬 레이어라 작업량 크다. 계절·기후 안정된 뒤에 손대라.

## 2. 구현 순서 & 통합 지점

시뮬 코어(`SimulationTick.LocalRules`)에 끼워넣는 방식이라 아키텍처 대변경 없음.

1. **계절** — `LocalRules` 진입부에서 계절 변조값 계산, 상수 대체. (하루면 됨)
2. **위도** — 셀 루프 안에서 `y`로 `tempFactor` 계산해 성장률에 곱함.
3. **물 흐름** — `Diffuse` 앞에 물 흐름 패스 추가 + `Elevation` 생성 로직.

각 단계는 독립적으로 켜고 끌 수 있게 config 플래그로. 하나씩 넣고 창발 확인.

## 3. 선택 트랙 A: GPU 이식 (하고 싶을 때, 마지막에)

**지금 하지 마라.** 이유:
- 로직(계절/위도/물흐름)이 계속 추가되는 중엔 GPU 커널을 매번 다시 짜야 함.
- 백만 셀은 "장관"이지 "더 나은 게임"이 아님. 출시엔 수만 셀로 충분.

**할 거라면** (환경 로직 다 굳은 뒤):
- SoA 배열을 그대로 `ComputeBuffer`에 올림. Phase 0에서 SoA로 짠 게 이걸 위한 포석.
- `LocalRules` → 커널1, `Diffuse`/물흐름 → 커널2. 더블 버퍼는 두 버퍼 ping-pong.
- 렌더는 `ComputeBuffer` → `RenderTexture` 직접 write(SetPixels 우회) → 대폭 빨라짐.
- 주의: 이산 틱을 유지하되 GPU에선 dispatch 한 번 = 한 틱.

## 4. 선택 트랙 B: 종 확장 (신중히, 최소로)

넣는다면 **한 번에 하나씩, 밸런싱 확인하며.** 후보:

| 종 | 역할 | 리스크 |
|---|---|---|
| 잡식 | 식물+초식 둘 다 먹음. 완충재 | 중 |
| 경쟁 식물 | 기존 식물과 자원 경쟁. 우점종 교체 드라마 | 중 |
| 기생 | 특정 종 밀도에 비례해 감소시킴 | 높음(불안정) |

- 3종 로트카-볼테라도 카오스로 빠지기 쉬운데 종이 늘면 튜닝 난이도 급증.
- **환경 시스템만으로 충분히 깊으면 이 트랙은 스킵해도 됨.** 완성·출시가 목표라면 특히.

## 5. Phase 2 완료 기준

- [ ] 계절이 돌고, 겨울에 개체군이 줄고 봄에 회복하는 **연 주기 흥망**이 관찰된다.
- [ ] (위도까지 했다면) 적도와 극지방의 운명이 공간적으로 갈린다.
- [ ] (물 흐름까지 했다면) 강가에 생명이 몰리고 고지대가 사막화된다.
- [ ] 위 전부 플레이어 개입 없이 자연 발생한다.

핵심은 첫 줄. **계절 하나로 "매년 반복되는 흥망"이 나오면 Phase 2의 8할은 성공.**
